# CWcomm 需求分析与实施计划（编码前）

## 1. 问题定义

在多语言活动现场，观众往往无法实时理解演讲内容。传统同传硬件方案成本高、部署重、扩展慢。CWcomm 目标是通过 Web 技术 + AI 实时语音能力，在用户自带设备上提供低门槛、多语种、可扩展的实时收听体验。

---

## 2. 场景与边界

### 2.1 典型场景
- 技术大会主会场、分会场。
- 企业内训、跨国发布会。
- 校园讲座、政府公开活动。

### 2.2 业务边界（MVP）
**包含**
- 单会场实时收音。
- 至少 1 种源语言 -> 多目标语言翻译。
- 网页端实时收听 + 字幕。

**不包含（后续）**
- 离线音频回放编辑。
- 多会场复杂调度。
- 人工译员协作台。

---

## 3. 需求拆解

### 3.1 功能需求（FR）
- FR-01：管理员可创建活动并配置目标语种。
- FR-02：系统可接入现场音频流并持续上传。
- FR-03：系统可对音频进行流式 ASR。
- FR-04：系统可将识别文本翻译为多语种。
- FR-05：终端用户可选择语种并播放对应译音。
- FR-06：终端用户可查看实时字幕（原文/译文）。
- FR-07：系统可处理断线重连并维持会话。
- FR-08：活动信息支持两种来源：本系统创建 + 气候周网站 OpenAPI 同步。
- FR-09：支持气候周网站用户登录集成（SSO），并完成本地用户档案映射。

### 3.2 非功能需求（NFR）
- NFR-01：端到端平均延迟目标 2~6 秒。
- NFR-02：支持 500 人级活动（MVP 参考值，可压测校准）。
- NFR-03：具备基础监控与告警（延迟、错误率、在线人数）。
- NFR-04：跨端可用（PC/手机/iPad）。
- NFR-05：隐私与安全（鉴权、加密、日志留存策略）。
- NFR-06：外部依赖稳定性（OpenAPI 同步失败可重试并具备降级策略）。

---

## 4. 信息架构与用户流程

### 4.1 用户端流程
1. 扫码/打开链接进入活动。
2. 选择目标语种。
3. 播放实时译音并查看字幕。
4. 网络异常时自动重连/降级字幕。

### 4.2 管理端流程
1. 创建活动并设置语种，或从气候周 OpenAPI 导入活动。
2. 启动采集端并确认音频输入。
3. 观察实时状态（在线人数、延迟、错误）。
4. 结束活动并导出统计。

### 4.3 登录与身份流程
1. 用户点击“使用气候周账号登录”。
2. 跳转气候周身份平台完成认证并返回授权码。
3. CWcomm 服务端换取 token，拉取用户基本信息并建立本地会话。
4. 根据活动角色映射授予听众或管理员权限。

---

## 5. 技术方案（候选）

## 5.1 方案 A：WebRTC 为主
- 优点：超低延迟、适合实时音频分发。
- 风险：浏览器兼容细节与 ICE/TURN 运维复杂。
- 适用：高实时要求、活动规模较大。

## 5.2 方案 B：HLS/LL-HLS + WebSocket
- 优点：架构相对简化。
- 风险：延迟可能较高，不利于“实时同传”体验。

## 5.3 建议路线
MVP 采用“WebRTC（音频）+ WebSocket（控制/字幕）+ OpenAPI/SSO 集成层”混合模式；如遇复杂网络环境可保留字幕优先兜底。

## 5.4 外部系统集成要求
- 活动同步：支持按活动 ID 拉取、按时间窗口增量同步、手动重同步。
- 认证授权：优先采用 OAuth2/OIDC，至少支持登录、登出、token 刷新。
- 数据映射：统一活动字段（名称、时间、地点、语种）和用户字段（唯一 ID、昵称、邮箱）。

---

## 6. 核心数据流（逻辑）

1. 采集端发送原始音频流。
2. 音频进入流式 ASR，产出增量文本。
3. 文本进入翻译服务，生成多语种片段。
4. 片段进入 TTS（可选）生成目标语种音频。
5. 分发层向终端推送音频与字幕。
6. 终端按语种订阅并播放。
7. 活动元数据定时与气候周 OpenAPI 同步，异常时告警并进入重试队列。

---

## 7. 风险清单与应对

- **延迟超标**：
  - 应对：缩短分片长度、并行翻译、模型 warm-up。
- **网络抖动**：
  - 应对：自适应缓冲、断线重连、字幕兜底。
- **噪声环境导致识别准确率下降**：
  - 应对：前置降噪、麦克风规范、术语热词。
- **成本不可控**：
  - 应对：语种按需启用、活动级限额与配额监控。
- **外部 API 不稳定或鉴权变更**：
  - 应对：设置缓存与重试、接口版本管理、预留手动导入与本地登录降级。

### M0：需求冻结（1 周）
- 交付：需求文档、成功指标、MVP 边界。

### M1：PoC（2 周）
- 交付：单语种端到端演示（音频+字幕），延迟基线。

### M2：MVP 开发（3~4 周）
- 交付：活动管理、语种切换、基础监控、重连机制。

### M3：试点与优化（2 周）
- 交付：真实活动复盘、稳定性修复、上线清单。

## 9. 编码前待确认事项

1. MVP 要支持的具体语种与优先级。
2. 目标并发规模与峰值时长。
3. 是否必须“译音”还是“字幕优先”。
4. 气候周 OpenAPI 的权限申请、限流规则、字段字典与 SLA。
5. 气候周用户登录集成协议（OAuth2/OIDC/SAML）与账号绑定策略。
6. AI 服务供应商（自研/第三方）与预算约束。
7. 合规要求（数据驻留、留存周期、审计要求）。
